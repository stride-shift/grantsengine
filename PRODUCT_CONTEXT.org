#+TITLE: Product Context — d-lab Grant Engine
#+DATE: 2026-06-19

* Purpose

Present-state product context: what the Grant Engine is, who uses it, the d-lab
domain it serves (organisation, funding model, programme catalogue, funder
relationships, proposal strategy), and the design decisions that are
*intentional* — so an audit reads this BEFORE flagging an intentional choice as a
bug. It documents the *product and its domain*, not the code structure — it
survives refactors.

Code-level mechanics for the funder-strategy engine and programme detection are
NOT here: they live in =src/data/quick_reference.org= (canonical). This file
holds the business/domain *meaning*; that file holds the *how*. Intentionally
dormant + deferred features live in =DEFERRED.org= (linked, not duplicated).

* What It Is

The d-lab Grant Engine is a React + Express app for managing a grant-funding
pipeline for *d-lab NPC*, a South African youth-skills non-profit. It moves a
grant opportunity through a staged pipeline (scout → vet → qualify → draft →
review → submit → await → won/lost) and uses AI to draft proposals, research
funders, scout new opportunities, and surface pipeline intelligence.

| AI use | Provider | Where |
|--------|----------|-------|
| Proposal drafting, funder research, scouting, pipeline intelligence | OpenAI (proxied through the Express backend) | =server/routes/ai.js= |
| Grant-scraper parsing/verification | Anthropic Claude (Haiku) | server scraper path |

* Proposal Strategy

The strategy a proposal takes is driven per-funder by =funderStrategy(grant)=
(returns lead / hook / language / structure / budget emphasis). Named-funder
intelligence lives in =FUNDER_ANGLES= (=src/data/funderAngles.js=); the
resolution mechanics are in =src/data/quick_reference.org=. The *business intent*
behind it:

| Situation | Framing |
|-----------|---------|
| Returning funder | "Here's what your investment built" + specific outcomes from the last cycle |
| New funder | "Here's the system and the proof" + matched programme type + an alumni story |

** Standing rules

- *Always include* the 30% org-cost contribution for sustainability.
- *Anti-patterns (hard rules):* never lead with "SA has X% unemployment"; never
  use hollow phrases; always use specific numbers. These are baked into the AI
  prompts (see Intentional Design Decisions).

* Intentional Design Decisions

Rationale for choices an audit might otherwise flag as defects. Each is stated as
*intentional: X, because Y* — present-state design, not a regression. (Code-level
quality findings remain valid issues; this section concerns only *deliberate
product/domain choices*.)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| ZAR-only currency | Currency is always South African Rand, formatted =R1,236,000= / =R1.2M=; no multi-currency support | d-lab operates entirely in ZAR; every funder, budget, and programme figure is Rand. Multi-currency would be unused complexity. Do not flag the single-currency assumption as a gap |
| Org context injected into every AI prompt | =CTX= / =CTX_SLIM= (=src/data/context.js=) — the d-lab profile, programme budgets, impact stats, funder history — is injected as =orgCtx= into every AI call | The prompts must share one version-controlled factual basis so the AI cannot invent or drift on numbers across proposals. It is CODE on purpose; changing a figure there changes what the AI may claim everywhere. =CTX_SLIM= is the token-budget variant of the same truth, not a second source |
| Anti-pattern guardrails baked into prompts | Every prompt carries the anti-pattern list (no "SA has X% unemployment" openers, no hollow phrases, always specific numbers) plus the tone contract (lead with the human story, weave numbers into narrative, emphasise the system, never lead with geographic expansion) | These are d-lab's house proposal voice and are load-bearing; the tone block in =context.js= is constraint, not decoration. Do not flag the repeated anti-pattern text as prompt bloat |
| Alumni-story "use once" rule | Prompts flag alumni stories "use each ONCE — do NOT fabricate" | Reusing or inventing beneficiary stories would damage funder trust and credibility; scarcity + honesty is deliberate |
| GIDF framed as continuity, never expansion | The cornerstone-funder hook for Get It Done Foundation is a continuity story, not a growth pitch | GIDF funds ~half the project budget; the relationship is protected by conservative, renewal-oriented framing. This is a domain rule, not a copy oversight |
| 8 programme types as a fixed catalogue | =PTYPES= is exactly 8 types with fixed budgets; Type 6 alone carries =cost: null= / =students: null= (priced per-learner) | The catalogue mirrors d-lab's actual programme offering — a fixed domain set, not an extensible config. Type 6's null cost is correct (per-learner pricing) and is deliberately skipped by the budget-fit loops; do not flag it as missing data |
| Funder lookup is exact-match | =FUNDER_ANGLES= is keyed by exact funder name, with aliases as explicit separate keys (e.g. "DGMT" + "DG Murray Trust"); no fuzzy matching | The funder set is small and curated; an unknown name deliberately falls to the no-intel hook (instructing the AI to research priorities rather than use generic mission language). Fuzzy matching would risk mis-attributing the wrong funder's strategy |
| 30% org-cost contribution always included | Proposals carry a standing 30% core-ops contribution line | Sustainability requirement of the funding model, not a per-proposal toggle |
