# d-lab Grant Engine — Executive Evaluation Report

_Generated by a 25-agent evaluation workflow: 7 dimensions reviewed in parallel, all high/critical findings adversarially verified (55 confirmed findings, 0 refuted)._

## 1. Executive Summary

The Grant Engine is a functional, thoughtfully-built tool for a small NPO, and it gets the hardest fundamentals right: every SQL query is parameterised and org-scoped, sessions use 256-bit tokens, passwords use bcrypt, the frontend has an ErrorBoundary with optimistic-rollback mutations, and ZAR/South-African formatting is consistently correct. However, the evaluation surfaced one **critical** issue and several **high**-severity problems that materially undermine security, core features, and deployability. Two flagship features are silently broken end-to-end — Google Calendar sync (invalid same-day all-day events) and starred reference-proposal injection (a `ReferenceError` swallowed by a `catch`) — while the deploy story is ambiguous enough that deadline reminders may never fire in production. Accessibility is weak throughout (almost no ARIA, keyboard-inoperable controls, sub-AA contrast).

**The single most important thing to fix:** the live Supabase personal access token committed to `mcp.json` (and present in git history) — it grants full management-API control of the production database holding all grant data and member PII. Revoke and rotate it immediately, then treat the DB as potentially compromised.

## 3. Per-Dimension Assessment

### Security (backend)
The fundamentals are strong where they matter most — parameterised, org-scoped queries (no injection), 256-bit session tokens, and bcrypt passwords. The serious exposure is concentrated in credentials and authorization: a leaked Supabase PAT, an unauthenticated account-takeover path, missing director checks on config writes, and a broad SSRF surface. For a small NPO the leaked credential and account-takeover are the highest real-world risk.
- *Medium* — Seeded default org password (`dlab2026`) is weak, predictable, and printed to startup logs; shared-password sessions have no `memberId` yet can still reach config writes (`server/index.js:53-57`).
- *Medium* — Auth token accepted via `req.query.token` (leaks into logs/history/Referer); `calendar.ics` feed is fully public to anyone who knows the org slug, exposing the entire pipeline (`server/middleware/auth.js:18`; `server/routes/data.js:303-391`).
- *Medium* — Upload `fileFilter` uses OR (mimetype **or** extension), making the mimetype check meaningless; SVGs allowed into the **public** logos bucket → stored XSS under the org's own domain (`server/routes/uploads.js:36-52`; `orgs.js:33-37,192`).
- *Medium* — `forgot-password` lets a single shared `SUPER_ADMIN_KEY` reset any member in any org to a known password and auto-login; `.env.example` default is `change-me-to-a-secure-key` (`server/routes/auth.js:208-249`).
- *Low* — Prompt-injection: scraped/uploaded content flows unsanitised into AI prompts and back into the pipeline; mapping prompt includes `bank_details`/`reg_numbers` (`server/scraper/scrapeGrants.js:303-313`; `autofill.js:75,110`; `uploads.js:138`).
- *Low* — Postgres TLS uses `rejectUnauthorized:false` for Supabase — encrypted but unverified, open to active MITM (`server/db.js:34`).

### Backend Correctness
Well-structured overall: async handlers forward errors, writes are parameterised, bulk-replace is transactional, and fire-and-forget side-effects are isolated. The high-impact bug is the broken Google Calendar all-day-event sync; remaining issues are logic gaps in the reminder/scout jobs, none data-corrupting.
- *Medium* — Custom gcal event IDs can be invalid base32hex (w/x/y/z disallowed), causing silent insert failures for future ID schemes (`server/routes/gcal.js:107-109,128-139`).
- *Medium* — 48-hour "deadline-missing" nudge never fires for manually-created grants because it relies on `log[0].d`, which is empty for POST/UI-created grants (`server/jobs/deadlineReminders.js:67-89`).
- *Medium* — Reminders depend on a 30-min cron hitting narrow 2-hour windows; on serverless/restarted hosts a missed run permanently drops the reminder (`deadlineReminders.js:33-64`; `server/index.js:76-81`).
- *Low* — Nightly scout comment/log says midnight SAST but cron actually fires at 22:00 SAST (`server/index.js:64-73`).
- *Low* — Scout funder-level dedupe blocks every future opportunity from any existing funder (incl. closed/lost) — suppresses exactly the returning-funder renewals the org cares about (`server/jobs/scout.js:164-167,181-199`).
- *Low* — `extractWithVision` ignores `response.ok`; non-OK OpenAI responses parse as "no text found", masking real API errors (`server/extractors/index.js:32-56`).
- *Low* — ICS feed and reminder times use server-local timezone, not SAST — reminders fire 2 hours off on a UTC host (`server/routes/data.js:307-312`; `deadlineReminders.js:33`).

### Frontend Correctness
Generally robust: top-level ErrorBoundary, AI calls return error strings (never throw) guarded by `isAIError`, optimistic mutations with rollback, and most AI-JSON parses wrapped in try/catch. Defects are localized — none white-screen the app, but several silently produce wrong UI or inconsistent data.
- *Medium* — Pipeline owner-filter avatars read non-existent fields `m.c`/`m.ini`, so every chip shows grey + the member ID's first two chars instead of initials (`src/components/Pipeline.jsx:570,575,577`).
- *Medium* — Background hygiene `persist()` rebuilds each server payload from the frozen `initialGrants` snapshot, so multi-pass touches overwrite earlier passes' fields in Supabase (UI stays correct) (`src/App.jsx:396-402,404-610`).
- *Medium* — Several AI-result handlers `JSON.parse` without an `isAIError()` pre-check, turning rate-limit messages into misleading "Could not parse" UX (`Pipeline.jsx:714`; `AutoFillPanel.jsx:121,317`; `App.jsx:521-528,602`).
- *Low* — `AutoFillPanel` form-detection effect has empty deps but reads `grant`/`job`/`isFormBased`, so auto-detection is order-dependent and may never fire (`src/components/AutoFillPanel.jsx:438-441`).
- *Low* — Scout badges render literal escape text instead of `●`/`⚠` glyphs (`src/components/ScoutPanel.jsx:694,700`).
- *Low* — View-log dedupe effect omits `grants` from deps; safe today but fragile and inconsistent with sibling effects (`src/App.jsx:347-365`).

### Architecture & Maintainability
Sound macro-architecture (lazy-loaded views, clean server route split, shared `app.js`), but debt is concentrated in three god-files plus dead code and near-zero test coverage. Stale CLAUDE.md is itself a hazard since it's injected as authoritative context for AI-assisted changes.
- *Medium* — `GrantDetail.jsx` is a ~2,900-line component with 32 hooks and no internal boundaries — the highest-risk file to edit (`src/components/GrantDetail.jsx:228-3126`).
- *Medium* — `useAI.js` holds ~25 inline prompts in a 1,318-line if-chain; `prompts.js` exists but is barely used and its header is inaccurate (`src/hooks/useAI.js:78-1474`).
- *Medium* — Dead code: `_DEAD_findApplyUrl_legacy` branch still shipped, and the 2,781-line `grant-engine.monolith.jsx` duplicate has drifted (`useAI.js:1277-1320`).
- *Medium* — Near-zero test coverage for a data-mutating app; the silent auto-archive/URL-rewrite hygiene job is entirely untested (`src/__tests__/utils.test.js`).
- *Medium* — A 320-line background hygiene side-effect (4 mutation passes) lives directly in `App.jsx`, untestable and entangled with rendering (`src/App.jsx:386-621`).
- *Medium* — Stale CLAUDE.md documents Gemini/Anthropic where code now uses OpenAI (`CLAUDE.md`).
- *Low* — Inconsistent dev-port config: 3000/3001/3010/3011 across vite, CORS default, and docs (`vite.config.js:6-13`).
- *Low* — Heavy prop drilling of grants/team/stages through every view; adding shared state means editing ~10 call sites (`src/App.jsx:1171-1290`).

### AI / Prompt Engineering
The Gemini→OpenAI swap is structurally sound on the wire (correct request shapes, valid models, correct Claude Haiku scraper path). But it left provider-specific language and dead URL-handling behind, and introduced the `refProposals` `ReferenceError` that silently kills the starred-reference feature.
- *Medium* — Search-grounded prompts demand strict JSON but the Responses API path never enables JSON mode; a stray bracket in a citation mis-parses and the nightly scout silently logs "no results" (`server/routes/ai.js:55-65,248-263`; `prompts.js:130`).
- *Medium* — Prompt-injection: scraped page text and existing funder names are interpolated verbatim, and auto-scout writes results ≥65 to the live pipeline with no human review (`server/scraper/scrapeGrants.js:303-313`; `scout.js:165-167,181-199`).
- *Medium* — Inconsistent retry coverage: server-side `callOpenAI` has no retry (a single 429 skips a whole market); `api.js` retries on Anthropic's dead 529 but not OpenAI's 500/503 (`server/routes/ai.js:125-176`; `scout.js:221-229`; `src/api.js:399-466`).
- *Low* — Prompts still tell the model it has "Google Search" post-swap, and all vertexaisearch redirect-filtering is now dead code (`src/prompts.js:111`; `useAI.js:1195,1250-1251,1292-1293`).
- *Low* — Stale "Gemini undershoots" comment drives a 2× word-budget inflation now mis-applied to OpenAI, overshooting section lengths and burning tokens (`useAI.js:597`).
- *Low* — `callClaude` never inspects `stop_reason` for `max_tokens`; a truncated scraper JSON array surfaces only as "0 usable" (`server/routes/ai.js:207-229`).

### Dependencies & Build/Deploy Config
The weakest dimension operationally: incompatible deploy configs, an unwired Vercel cron, a misleading `.env.example`, dead animation libraries, and an out-of-sync lockfile that blocks `npm ci` and audits.
- *Medium* — `api/cron/reminders.js` exists but `vercel.json` has no `"crons"` array, so on Vercel deadline reminders never fire automatically (`vercel.json:1-17`; `api/cron/reminders.js:10-28`).
- *Medium* — Both `framer-motion` and `motion` declared (motion already bundles framer-motion); neither is imported — pure dead weight (`package.json:26,31`).
- *Medium* — `package-lock.json` out of sync with `package.json`; `npm audit` refuses to run and `npm ci` will error (`package-lock.json`, `package.json`).
- *Medium* — Port numbers inconsistent across vite (3010/3011), `.env.example` (3001), CORS default (3000), and docs (3010) — dev API proxy/CORS fail out of the box (`vite.config.js:7,10`; `.env.example:12`; `server/app.js:23`).
- *Low* — Root `start` script uses bash-only `mkdir -p`/`NODE_ENV=` syntax (fails on Windows); no `engines` field despite Vite 7's Node 20.19+ requirement (`package.json:9,15-45`; `.node-version:1`).
- *Low* — `playwright-service/` has no `.gitignore`; its `node_modules` is spared only by the root pattern (`.gitignore:1`).

### UX & Accessibility
UX has genuine strengths — collapsible sections, empty states, multi-step AI progress feedback, debounced search, bulk-archive confirmation, and correct/consistent ZAR formatting. Accessibility, however, is weak across the board: almost no ARIA, no semantic dialogs, keyboard-inoperable controls, unannounced toasts, and sub-AA contrast.
- *Medium* — Toasts have no `role`/`aria-live`, so success/error/undo feedback is never announced; icons are bare glyphs and the close button has no label (`src/components/Toast.jsx:48-100`).
- *Medium* — Tutorial and tour modals lack `role="dialog"`/`aria-modal`, focus management/trapping, and Escape-to-close — and the tutorial is the first thing a screen-reader user hits (`src/components/WelcomeTutorial.jsx:88-172`).
- *Medium* — Clickable `div`/`span` elements (upload drop zone, grant-name, dashboard cards, programme tiles) are not keyboard-operable (`src/components/UploadZone.jsx:101-116`).
- *Medium* — Destructive upload-delete and single-card Vetting reject fire with no confirmation/undo (the bulk paths confirm — inconsistent) (`src/components/UploadZone.jsx:87-94`; `Vetting.jsx:454-460`).
- *Medium* — Form inputs across the wizard, UploadZone, Vetting, and Settings rely on placeholders instead of associated labels (`src/components/Pipeline.jsx:794-831`).
- *Low* — Batch/OAuth operations give thin feedback: Score-All swallows per-grant failures to console; multi-file upload shows no N-of-M counter (`src/components/Pipeline.jsx:366-385`).
- *Low* — Decorative emoji/status glyphs aren't `aria-hidden`, and some state is conveyed by colour alone (`src/components/Vetting.jsx:7-12`).

## 4. Top 5 Recommended Actions (ordered by leverage)

1. **Revoke and rotate the leaked Supabase token, then purge it from history.** (`mcp.json`) Minutes of effort, eliminates the only critical finding and the single largest real-world risk. Also rotate the service-role key and treat the DB as potentially compromised.

2. **Close the account-takeover and missing-director-authz gaps.** (`server/routes/auth.js:139-176`; `server/routes/orgs.js:92,108,126,178`) Low effort (add `requireDirector`, gate first-password-set behind the existing reset-token flow, reject `memberId`-less sessions on config writes) for high security payoff — these are the paths to full admin compromise.

3. **Fix the two silently-dead features with one-line root causes.** (`useAI.js:234` and `gcal.js:111-126`) Tiny effort, high user-visible impact: restore starred-reference injection into every draft (the app's core IP) and make Google Calendar sync actually create events instead of reporting false success.

4. **Make the deploy story coherent and reminders actually fire.** Pick one platform, delete the other configs, wire the Vercel `crons` array (or guarantee a long-lived worker for node-cron), regenerate the lockfile (`npm install` → verify `npm ci`), and rewrite `.env.example` to match the code (Resend + Google + CRON_SECRET, drop SMTP). Moderate effort; without it, deadline reminders — the tool's core value — never go out in production.

5. **Add an SSRF guard and start the a11y/contrast cleanup.** Wrap all server-side fetches of user URLs in a private-range/scheme guard (one shared helper covers scraper, autofill, verify-urls, favicon), and darken `C.t4` to ~#6B7280 plus add `role`/`aria-live` to toasts and labels/`role="button"` to the worst offenders. Each is contained, broadly applicable, and removes a class of risk rather than a single instance.
