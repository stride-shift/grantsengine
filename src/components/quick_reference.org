* Overview

The React component layer for the d-lab Grant Engine: UI primitives (defined in
=index.jsx=), major views (Dashboard / Pipeline / GrantDetail / Calendar /
Funders / Archive / Admin / Settings / DocVault / Freebies), and the AI-feature
panels (Scout, AutoFill, ProposalWorkspace, etc.). All components are
*presentational* — application state lives in =App.jsx= and is passed down;
mutations route through prop helpers (see Key Patterns). No CSS framework: every
style is inline, sourced from the =C= token object in =src/theme.js=.

** =*Parts.jsx= Extractions

Each =*Parts.jsx= holds sub-components lifted OUT of its same-named parent. They
are *module-scope, props-only presentational* pieces — extracted so the parent
stays readable and (critically) so the sub-components keep *stable React
identity* (see Constraints). Logic and AI state stay in the parent.

* Key Patterns

** Presentational / State-Down Contract

Components hold only *local UI state* (open/closed, active tab, edit buffers).
All shared/grant state lives in =App.jsx= and is threaded down as props. The
mutation helpers are also props — components never call the API directly:

| Helper | Shape | Effect |
|--------|-------|--------|
| =up= | =up(id, { field: value })= | Updates a grant → PUT → Supabase |
| =addL= | =addL(id, "text")= | Appends an activity-log entry |
| =sB= | =sB("key", true/false)= | Sets an AI busy flag |
| =sA= | =sA("key", value)= | Stores an AI result |

*Gotcha*: inside =GrantDetail= the threaded =up= is *pre-bound to the current
grant*, so it is called =up(field, value)= (2-arg) there — NOT =up(id, changes)=.
The 2-arg form is grant-scoped sugar local to that view; the canonical app-level
helper is the 3-arg =up(id, changes)=.

** AI Data Flow (panels → state → persistence)

AI panels invoke calls through =src/hooks/useAI.js= (see =hooks/quick_reference.org=
for prompt/logic detail — not duplicated here). The UI contract:

- Busy state written via =sB()=, result via =sA()=; =AICard= / =AILoadingPanel=
  render the busy/result/error states from those.
- Durable AI output is persisted onto the grant via =up()= (e.g.
  =aiResearch=, =aiSections=, =aiFitscore=).
- Session-scoped / cross-grant AI memory (scout brief, scout rejections,
  proposal references) is stored via the KV helpers (=kvGet= / =kvSet=), NOT on
  the grant record.
- =AddGrantWizard= does not run AI itself: it stamps =_pendingAI= on the new
  grant and =App.jsx= sequences the fit-score/research/draft runs afterward.

** Design Tokens (=src/theme.js= =C= object)

No CSS framework. Every colour/surface comes from the =C= token object imported
from =src/theme.js=; styling is inline =style={{…}}=. The token table itself is
canonical in =src/theme.js= and is not reproduced here. Exceptions: the dark auth
screens (=Login=) and a few canvas effects use literal colours rather than =C=.
Animation strategy: CSS transitions for chrome (tour, buttons, tooltips),
=<style>=-injected =@keyframes= for small effects, and =<canvas>= (NorthernLights,
SpiralAnimation) for the heavy backgrounds.

** Funder / Programme Intelligence in the UI

- =Funders= resolves a relationship owner two-tier: user-assigned KV
  (=funder_owners=) wins over the hardcoded =REL_OWNERS= default; it calls
  =funderStrategy(grant)= on a representative active grant for the strategy panel.
- =BudgetBuilder= loads line items from =PTYPES[type].table= (the 8 real d-lab
  programme types) and applies cohort/year multipliers; the 30% org-contribution
  toggle and funder-budget utilisation warnings live here.

* Constraints & Gotchas

- *Stable-identity rule*: =SectionWrap= (in =GrantDetail=) and the =Section= /
  chart helpers (in =DashboardParts=) are deliberately defined at *module scope*,
  never inside the parent render. Redefining them per-render gives them a new
  identity each frame, which remounts children and wipes their local state
  (e.g. ProposalWorkspace's preview toggle). This is why the =*Parts= extractions
  exist as standalone module functions.
- =up()= is 3-arg (=up(id, changes)=) app-wide but *2-arg grant-bound*
  (=up(field, value)=) inside =GrantDetail=. Do not mix them.
- =ScoutPanel=: AI results with no URL are auto-filtered (hallucination shield);
  grounding-redirect URLs are resolved before display; SA + Global searches are
  *sequenced* (not parallel) to avoid hitting the model RPM limit; falls back to
  =SCOUT_FALLBACK= if JSON parse fails; dedup uses normalised funder name + fuzzy
  title match.
- =AutoFillPanel=: form-field detection is NOT auto-triggered when the apply-URL
  is missing — it is an explicit user action; the =generatingProposal= prop
  auto-generates the proposal on first open (guarded by a useRef).
- =ProposalWorkspace=: =generateAll= keeps a local =currentSections= copy while
  persisting after each section to dodge a stale-closure bug; a recommended ask
  is parsed from the budget section but is NEVER forced when =askSource= is
  user-override / manual / budget-builder.
- =BudgetBuilder=: the ask is not computed inline — it is written explicitly on
  Save (=askSource="budget-builder"=); clearing the budget wipes =budgetTable= +
  =askYears=, not just zeroes them.
- =DocVault=: AI-reference IDs and category changes use non-CRUD endpoints
  (KV store; =PUT …/uploads/:id/category=; =POST …/uploads/:id/reextract= as a
  background text-extraction job).
- =Toast= is a context provider (=useToast()=); max 5 toasts, oldest evicted;
  timers held in refs and cleared on unmount.
- =AddGrantWizard=: form state persists across modal open/close (gated by the
  =open= prop, not conditional mount); queued files upload only AFTER the grant
  is created.
- =Settings= has no AI itself, but its knowledge-base content is loaded into
  every AI prompt as =orgCtx= (via =useAI=) — editing it changes all AI output.
- =Dashboard= reorders sections visually via CSS flex =order= (negative values),
  not by moving JSX; intelligence panels gate on a minimum grant count.
